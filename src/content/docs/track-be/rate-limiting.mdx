## 1. Conceptos

El rate limiting protege el API de dos tipos de abuso: ataques intencionales (bots, scrapers) y errores de código en el cliente (loops que hacen cientos de requests por segundo sin darse cuenta).

Un solo nivel de rate limiting no es suficiente. Si el límite es solo por IP, un atacante usa múltiples IPs. Si es solo por tenant, un tenant malintencionado puede hacer muchas requests. Rush usa tres niveles jerárquicos.

### Los tres niveles

**Nivel 1 — Global por IP**: 200 requests por minuto desde la misma IP, sin importar qué tenant o usuario. Protege contra bots que no tienen autenticación.

**Nivel 2 — Por tenant**: 1.000 requests por minuto por tenant. Un tenant no puede saturar el API aunque use múltiples usuarios.

**Nivel 3 — Por usuario**: 100 requests por minuto por usuario individual. Protege contra usuarios que tienen un bug en su código de integración.

### La implementación con Redis

```ts
// src/common/guards/rate-limit.guard.ts
import { Injectable, CanActivate, ExecutionContext } from '@nestjs/common';
import { TooManyRequestsException } from '@nestjs/common';
import Redis from 'ioredis';

interface RateLimitConfig {
  key: string;
  limit: number;
  windowSeconds: number;
}

@Injectable()
export class RateLimitGuard implements CanActivate {
  constructor(private readonly redis: Redis) {}

  private async checkLimit(config: RateLimitConfig): Promise<boolean> {
    const { key, limit, windowSeconds } = config;
    const count = await this.redis.incr(key);
    if (count === 1) {
      await this.redis.expire(key, windowSeconds);
    }
    return count <= limit;
  }

  async canActivate(context: ExecutionContext): Promise<boolean> {
    const req = context.switchToHttp().getRequest<{
      ip: string;
      user?: { id: string; businessId: string };
    }>();

    const now = Math.floor(Date.now() / 60000);

    const globalOk = await this.checkLimit({
      key: `rl:global:${req.ip}:${now}`,
      limit: 200,
      windowSeconds: 60,
    });

    if (!globalOk) {
      throw new TooManyRequestsException('Global rate limit exceeded');
    }

    if (req.user?.businessId) {
      const tenantOk = await this.checkLimit({
        key: `rl:tenant:${req.user.businessId}:${now}`,
        limit: 1000,
        windowSeconds: 60,
      });

      if (!tenantOk) {
        throw new TooManyRequestsException('Tenant rate limit exceeded');
      }
    }

    if (req.user?.id) {
      const userOk = await this.checkLimit({
        key: `rl:user:${req.user.id}:${now}`,
        limit: 100,
        windowSeconds: 60,
      });

      if (!userOk) {
        throw new TooManyRequestsException('User rate limit exceeded');
      }
    }

    return true;
  }
}
```

### El header Retry-After

Cuando el límite se alcanza, el API debe incluir el header `Retry-After` con cuántos segundos debe esperar el cliente:

```ts
if (!globalOk) {
  const ttl = await this.redis.ttl(`rl:global:${req.ip}:${now}`);
  const response = context.switchToHttp().getResponse<{ setHeader: (key: string, value: string) => void }>();
  response.setHeader('Retry-After', String(ttl));
  throw new TooManyRequestsException('Global rate limit exceeded');
}
```

### Dónde no poner rate limiting

El guard global va en todos los endpoints protegidos. Pero para el endpoint de health check (`/health`) y para el endpoint de login, usa límites distintos:

- `/health`: sin rate limit (necesita responder siempre para load balancers)
- `/auth/sign-in`: rate limit estricto de 5 por minuto por IP para prevenir brute force

## 2. Lab guiado

<LabSpec
  title="Rate limiting de tres niveles con Redis"
  estimatedMinutes={35}

Vas a implementar el guard de rate limiting y verificar que cada nivel funciona independientemente.

### Paso 1: implementar el guard

Crea `src/common/guards/rate-limit.guard.ts` con el código del ejemplo.

### Paso 2: aplicar el guard globalmente

En `app.module.ts` o `main.ts`, registra el guard globalmente:

```ts
app.useGlobalGuards(new RateLimitGuard(redis));
```

### Paso 3: test de rate limiting

Baja los límites temporalmente para el test:

```ts
const LIMITS = {
  global: { limit: 5, windowSeconds: 60 },
  tenant: { limit: 10, windowSeconds: 60 },
  user: { limit: 3, windowSeconds: 60 },
};
```

Envía 4 requests con el mismo usuario:

```bash
for i in {1..4}; do
  curl -s -o /dev/null -w "%{http_code}\n" \
    http://localhost:3000/sales/biz-123 \
    -H "Content-Type: application/json" \
    -d '{"amount": 1}'
done
```

Los primeros 3 deben devolver 201. El 4to debe devolver 429.

### Verificación final

Verifica en Redis que la key de rate limit existe y tiene TTL:

```bash
docker exec -it lab-redis redis-cli TTL "rl:user:<user-id>:<window>"
```

Debe devolver un número entre 1 y 60.
