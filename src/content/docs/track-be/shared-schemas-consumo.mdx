## 1. Conceptos

Rush tiene un paquete compartido, `@rush/shared-schemas`, que contiene los schemas Zod que tanto el backend como el frontend necesitan conocer. El schema de una venta, el schema de un business, el schema de una respuesta de error — esos van ahí.

La razón de tener un paquete compartido: si el backend cambia el formato de una respuesta sin coordinar con el frontend, el frontend se rompe silenciosamente. Con un paquete compartido con semver, ese cambio es un breaking change explícito.

### Qué va en shared-schemas

**Sí va en shared-schemas**:

- Schemas de request y response del API que el frontend consume
- Tipos compartidos (enums de moneda, tipos de evento, estados de business)
- Errores tipados que el frontend puede parsear

**No va en shared-schemas**:

- Schemas internos del backend (validaciones de use case que el frontend no ve)
- Schemas de base de datos (Drizzle schema)
- Schemas de configuración del servidor

### Importar del paquete

```ts
// src/sales/sales.controller.ts
import { createSaleRequestSchema, CreateSaleRequest } from '@rush/shared-schemas';
import { ZodValidationPipe } from '../common/pipes/zod-validation.pipe';

@Controller('sales')
export class SalesController {
  @Post(':businessId')
  create(
    @Param('businessId') businessId: string,
    @Body(new ZodValidationPipe(createSaleRequestSchema)) dto: CreateSaleRequest,
  ) {
    return this.salesService.recordSale({ businessId, ...dto });
  }
}
```

El tipo `CreateSaleRequest` es inferido del schema — no hay que definirlo dos veces.

### Manejar breaking changes

Si el `createSaleRequestSchema` cambia de forma, todos los consumidores (backend y frontend) deben actualizarse. El flujo es:

1. Hacer el cambio en el schema con un changeset que indica si es `major` (breaking) o `minor/patch`
2. Publicar la nueva versión del paquete
3. Actualizar los consumidores uno a uno

```bash
pnpm changeset
```

El CLI pregunta qué tipo de cambio es. Para un campo nuevo opcional: `minor`. Para renombrar un campo: `major`.

### Versión pinneada en el backend

```json
{
  "dependencies": {
    "@rush/shared-schemas": "1.2.0"
  }
}
```

Versión exacta, sin `^`. Eso garantiza que un `pnpm update` accidental no actualiza automáticamente el paquete y rompe el backend.

Para actualizar, es un cambio explícito de versión en `package.json` con un PR dedicado.

### Compatibilidad hacia atrás

Cuando agregas un campo al schema, agrégalo como opcional primero:

```ts
// En @rush/shared-schemas — cambio sin breaking
export const createSaleRequestSchema = z.object({
  amount: z.number().positive(),
  currency: z.enum(['VES', 'USD']),
  description: z.string().optional(),
});
```

Todos los consumidores que no mandan `description` siguen funcionando. Cuando todos los consumidores estén actualizados, puedes hacerlo requerido en el siguiente major.

## 2. Lab guiado

<LabSpec
  title="Simular @rush/shared-schemas con un paquete local"
  estimatedMinutes={30}

>

Como el paquete real está en el monorepo de Rush, vas a simular el paquete con un directorio local para el lab.

### Paso 1: crear el paquete local

```bash
mkdir -p packages/shared-schemas/src
cd packages/shared-schemas
pnpm init
```

Edita el `package.json`:

```json
{
  "name": "@rush/shared-schemas",
  "version": "1.0.0",
  "main": "src/index.ts",
  "types": "src/index.ts"
}
```

### Paso 2: definir los schemas compartidos

```ts
// packages/shared-schemas/src/index.ts
import { z } from 'zod';

export const createSaleRequestSchema = z.object({
  amount: z.number().positive().max(999_999_999),
  currency: z.enum(['VES', 'USD']),
  idempotencyKey: z.string().uuid(),
});

export type CreateSaleRequest = z.infer<typeof createSaleRequestSchema>;

export const saleResponseSchema = z.object({
  id: z.string().uuid(),
  businessId: z.string().uuid(),
  amount: z.string(),
  currency: z.string(),
  createdAt: z.string(),
});

export type SaleResponse = z.infer<typeof saleResponseSchema>;
```

### Paso 3: vincularlo al proyecto principal

En el `package.json` del proyecto NestJS:

```json
{
  "dependencies": {
    "@rush/shared-schemas": "workspace:*"
  }
}
```

```bash
pnpm install
```

### Paso 4: usar el schema en el controller

Importa `createSaleRequestSchema` del paquete y úsalo en el pipe del controller.

### Verificación final

El controller valida el body con el schema del paquete compartido. Si cambias el schema en el paquete (por ejemplo, agregas un campo requerido), TypeScript en el controller se queja hasta que actualices el código del controller. Eso es el contrato funcionando.
