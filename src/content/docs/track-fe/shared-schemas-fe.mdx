## 1. Conceptos

### Por qué compartir schemas entre frontend y backend

Sin `@rush/shared-schemas`, el frontend y el backend definen el mismo contrato en dos lugares:

```ts
// backend: src/sales/dto/create-sale.dto.ts
const createSaleSchema = z.object({ amount: z.number().positive() });

// frontend: src/features/sales/schemas/sale.schema.ts
const saleSchema = z.object({ amount: z.number().positive() });
```

Cuando alguien cambia el schema en el backend y se olvida de actualizar el frontend, los tipos divergen silenciosamente. El error aparece en runtime, no en compilación.

`@rush/shared-schemas` es el package que vive en el monorepo de Rush y contiene los schemas canónicos. Tanto el backend como el frontend lo consumen. Un solo lugar de verdad.

### Validar responses del API en el frontend

La mayoría de los devs asumen que si el backend devuelve `200 OK`, los datos tienen la forma correcta. Eso no es cierto si:

- El backend tiene un bug y devuelve campos extra o faltantes
- Hay una versión desactualizada del backend en staging
- Un interceptor en medio modificó la respuesta

Validar el response con el schema de Zod en el cliente lo detecta antes de que el dato corrupto llegue a los componentes:

```ts
const data = await response.json();
const parsed = saleSchema.parse(data);
```

Si `parse` lanza, el error se propaga al `queryFn` y TanStack Query lo captura como error de la query — el componente muestra el estado de error en vez de datos corruptos.

### Versiones del package y breaking changes

`@rush/shared-schemas` usa semver:

- `patch`: fix de typo en mensaje de error, sin cambio de tipos
- `minor`: campo nuevo opcional — retrocompatible
- `major`: campo renombrado, tipo cambiado — breaking change

El frontend debe pinear la versión de `@rush/shared-schemas` y actualizar deliberadamente, no con `^`:

```json
{
  "dependencies": {
    "@rush/shared-schemas": "1.2.0"
  }
}
```

Con `^1.2.0`, pnpm puede instalar `1.3.0` silenciosamente y el frontend puede romper si `1.3.0` tiene un minor que cambia el tipo de un campo opcional.

## 2. Lab guiado

<LabSpec title="Consumir @rush/shared-schemas en el frontend" estimatedMinutes={40} runnable={false}>

Vas a configurar el consumo del package compartido y validar el response del API de ventas con el schema canónico.

### Paso 1: instalar el package

Asumiendo que el monorepo de Rush tiene `@rush/shared-schemas` publicado en GitHub Packages:

```bash
pnpm add @rush/shared-schemas@1.0.0
```

Si estás trabajando localmente con el monorepo, usa workspace linking:

```bash
pnpm add @rush/shared-schemas@workspace:*
```

### Paso 2: usar el schema en la capa de API del frontend

```ts
// src/features/sales/api/sales.api.ts
import { saleSchema } from '@rush/shared-schemas';
import { z } from 'zod';

const salesListSchema = z.array(saleSchema);

export async function fetchSales(businessId: string) {
  const response = await fetch(`/api/businesses/${businessId}/sales`);

  if (!response.ok) {
    throw new Error(`API error: ${response.status}`);
  }

  const raw = await response.json();

  const parsed = salesListSchema.safeParse(raw);

  if (!parsed.success) {
    console.error('Unexpected API response shape:', parsed.error.format());
    throw new Error('Invalid response from server');
  }

  return parsed.data;
}
```

### Paso 3: tipos derivados del schema compartido

```ts
// src/features/sales/schemas/index.ts
import { saleSchema } from '@rush/shared-schemas';
import { z } from 'zod';

export type Sale = z.infer<typeof saleSchema>;

export const createSaleRequestSchema = saleSchema.pick({
  amount: true,
  currency: true,
}).extend({
  businessId: z.string().uuid(),
});

export type CreateSaleRequest = z.infer<typeof createSaleRequestSchema>;
```

### Paso 4: verificar con TypeScript que los tipos coinciden

```ts
// Esto debe compilar sin errores
const sale: Sale = {
  id: 'abc-123',
  businessId: 'biz-001',
  amount: 150.5,
  currency: 'VES',
  usdEquivalent: 4.18,
  createdAt: '2026-05-18T10:30:00Z',
};
```

Si el schema del package cambia (por ejemplo, agrega un campo `referenceRate` requerido), TypeScript te avisará que el objeto de arriba está incompleto.

### Verificación final

```bash
pnpm tsc --noEmit
```

Debe salir sin errores. Si `@rush/shared-schemas` tiene un campo que tu código no está usando, TypeScript no se queja — los campos extra se ignoran. Si tiene un campo requerido que tu código no provee, TypeScript falla.
