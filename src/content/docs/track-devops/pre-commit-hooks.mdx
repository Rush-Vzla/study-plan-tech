## 1. Conceptos

**1. ¿Para qué sirven los pre-commit hooks?**

Un pre-commit hook es un script que Git corre automáticamente antes de crear el commit. Si el script falla, el commit no se crea.

Esto te da una red de seguridad local: el código formateado, lintado y sin secretos llega al repositorio desde el principio, antes de que el CI lo detecte. Es mucho más barato arreglar un problema local que arreglarlo en una PR abierta.

Acá está el punto: los hooks no reemplazan al CI. El CI es la validación autoritativa. Los hooks son una conveniencia para el desarrollador.

**2. ¿Por qué `simple-git-hooks` y no Husky?**

Husky es más conocido, pero tiene más configuración y cambia cada versión mayor. `simple-git-hooks` es una alternativa minimalista: define tus hooks en `package.json` y listo. No tiene dependencias transitivas complicadas.

**3. ¿Qué es gitleaks y qué detecta?**

Gitleaks es una herramienta de escaneo de secretos. Revisa los archivos que vas a commitear buscando patrones que se parecen a credenciales: tokens de API, claves de AWS, cadenas de conexión de base de datos, claves privadas RSA.

Si detecta un match, bloquea el commit y te muestra qué encontró y en qué archivo. Eso te da la oportunidad de rotar el secreto antes de que llegue al historial de Git.

### Instalar las dependencias

```bash
pnpm add -D simple-git-hooks lint-staged
```

Gitleaks se instala como binario del sistema, no como dependencia de npm:

```bash
# macOS
brew install gitleaks

# Linux (CI o Droplet)
curl -Lo gitleaks.tar.gz https://github.com/gitleaks/gitleaks/releases/download/v8.21.2/gitleaks_8.21.2_linux_x64.tar.gz
tar -xzf gitleaks.tar.gz gitleaks
sudo mv gitleaks /usr/local/bin/
```

### Configurar en package.json

```json
{
  "simple-git-hooks": {
    "pre-commit": "pnpm lint-staged && gitleaks protect --staged --redact"
  },
  "lint-staged": {
    "*.{ts,tsx}": ["prettier --write", "eslint --fix"],
    "*.{json,yaml,yml,md}": ["prettier --write"]
  }
}
```

El `--staged` le dice a gitleaks que solo revise los archivos en el staging area, no el historial completo. El `--redact` reemplaza los secretos detectados en el output por `REDACTED` para que no aparezcan en los logs del terminal.

### Activar los hooks

```bash
pnpm simple-git-hooks
```

Esto escribe los scripts en `.git/hooks/pre-commit`. Cada vez que alguien hace `pnpm install`, `simple-git-hooks` reinstala los hooks automáticamente si están en `postinstall`.

### Agregar al package.json scripts

```json
{
  "scripts": {
    "postinstall": "simple-git-hooks"
  }
}
```

Así cualquier dev que clone el repo y haga `pnpm install` tiene los hooks configurados sin pasos manuales.

### Archivo .gitleaks.toml para excepciones

```toml
[allowlist]
  description = "Known safe patterns in this repo"
  paths = [
    "docs-internal/style-guide-maracucho.md",
  ]
  regexes = [
    "example\\.com"
  ]
```

Si gitleaks detecta falsos positivos, los documenta en `.gitleaks.toml`. Cada excepción debe tener una descripción que justifique por qué no es un secreto real.
