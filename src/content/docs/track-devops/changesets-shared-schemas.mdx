## 1. Conceptos

**1. ¿Qué es Changesets y qué problema resuelve?**

`@changesets/cli` es una herramienta para gestionar versiones de paquetes en un monorepo. El problema que resuelve es: ¿cómo decides si un cambio es un `patch`, un `minor` o un `major`? ¿Y cómo lo comunicas al equipo?

Con Changesets, cada vez que haces un cambio que impacta a los consumidores del paquete, creas un archivo de changeset. Ese archivo es parte del commit y describe qué cambió y con qué impacto en semver.

Cuando llega el momento de publicar, Changesets lee todos los archivos de changeset acumulados, determina la nueva versión, actualiza el `package.json`, genera el `CHANGELOG.md` y crea el tag de git.

**2. ¿Qué es `@rush/shared-schemas` y quién lo consume?**

`@rush/shared-schemas` es el paquete que contiene los schemas Zod compartidos entre el backend y el frontend de Rush. Cosas como el schema de una venta, el schema de un negocio, los tipos de respuesta de la API.

Lo consume:

- El backend de NestJS (para validar requests y tipar responses).
- El frontend de React (para validar responses de la API y tipar formularios).

Si cambias un schema en el paquete y alguno de los consumidores no actualiza, pueden aparecer errores en producción difíciles de debuggear.

**3. ¿Cómo funciona el bot de Changesets?**

El bot de Changesets corre en GitHub Actions. Cuando detecta archivos de changeset en un PR o en `main`, abre automáticamente un "Version PR" — un pull request que muestra exactamente qué versión va a publicarse y qué incluye el changelog.

Cuando ese PR se mergea, el bot publica el paquete a GitHub Packages con la versión calculada. No hay intervención manual.

### Instalar Changesets

```bash
pnpm add -D @changesets/cli
pnpm changeset init
```

Esto crea el directorio `.changeset/` con un archivo de configuración.

### Configurar para GitHub Packages

```json
{
  "name": "@rush/shared-schemas",
  "version": "1.0.0",
  "publishConfig": {
    "registry": "https://npm.pkg.github.com",
    "access": "restricted"
  }
}
```

```json
// .changeset/config.json
{
  "$schema": "https://unpkg.com/@changesets/config@3.0.0/schema.json",
  "changelog": "@changesets/cli/changelog",
  "commit": false,
  "fixed": [],
  "linked": [],
  "access": "restricted",
  "baseBranch": "main",
  "updateInternalDependencies": "patch",
  "ignore": []
}
```

### Crear un changeset al modificar el schema

Cuando agregas un campo nuevo al schema de ventas:

```bash
pnpm changeset
```

El CLI te hace preguntas interactivas:

1. ¿Qué paquetes cambiaron? → `@rush/shared-schemas`
2. ¿Qué tipo de cambio? → `minor` (nuevo campo opcional) o `major` (campo obligatorio que rompe consumidores)
3. ¿Describe el cambio? → "Add payment_method field to SaleSchema"

Esto crea un archivo en `.changeset/nombre-random.md`. Commitealo con el cambio.

### GitHub Actions para publicar

```yaml
name: Release

on:
  push:
    branches: [main]

jobs:
  release:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      packages: write
      pull-requests: write
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: pnpm/action-setup@v4
        with:
          version: 10

      - uses: actions/setup-node@v4
        with:
          node-version: '24'
          cache: 'pnpm'
          registry-url: 'https://npm.pkg.github.com'

      - run: pnpm install --frozen-lockfile

      - uses: changesets/action@v1
        with:
          publish: pnpm changeset publish
          version: pnpm changeset version
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NODE_AUTH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

El `changesets/action` hace todo: si hay changesets pendientes, abre el Version PR. Si el Version PR se mergea, publica el paquete.

### Actualizar el consumidor después de una publicación

En el repo del backend o frontend, después de que se publica una nueva versión:

```bash
pnpm update @rush/shared-schemas
```

Si es un breaking change (mayor), hay que revisar los places donde se usa el schema y actualizar.
