## 1. Conceptos

**1. ¿Qué escanea Trivy exactamente?**

Trivy analiza el contenido de una imagen Docker buscando paquetes del sistema operativo y dependencias de lenguaje con CVEs conocidos. Compara las versiones instaladas contra bases de datos de vulnerabilidades (NVD, GitHub Advisory Database, bases propias de cada distro).

Para una imagen distroless de Node, Trivy revisa:

- Los paquetes del sistema de la imagen base (muy pocos en distroless).
- Los paquetes de npm en `node_modules/`.

El resultado es una lista de CVEs clasificados por severidad: UNKNOWN, LOW, MEDIUM, HIGH, CRITICAL.

**2. ¿Cuándo bloquea el pipeline y cuándo no?**

En Rush, el pipeline bloquea si hay vulnerabilidades CRITICAL. Las HIGH se reportan pero no bloquean automáticamente — el equipo decide si son explotables en el contexto de Rush.

Esta diferencia existe porque CRITICAL generalmente significa que hay exploits públicos disponibles y el vector de ataque es remoto y sin autenticación. HIGH puede ser serio pero a menudo requiere condiciones específicas para ser explotado.

**3. ¿Qué es un `.trivyignore` y cuándo tiene sentido usarlo?**

A veces hay un CVE que sabes que no aplica a tu caso. Por ejemplo: una vulnerabilidad en una librería que Rush usa pero no en la funcionalidad afectada. Puedes ignorarla con `.trivyignore`, pero con una condición: debes documentar por qué la estás ignorando.

Un `.trivyignore` sin comentarios es una deuda de seguridad silenciosa. Con comentarios, es una decisión explícita y auditable.

### Escaneo básico de una imagen en CI

El step de Trivy en el pipeline de GitHub Actions:

```yaml
- uses: aquasecurity/trivy-action@master
  with:
    image-ref: ghcr.io/${{ github.repository }}:latest
    format: table
    exit-code: '1'
    severity: 'CRITICAL'
    trivyignores: .trivyignore
```

El `exit-code: '1'` hace que el step falle si encuentra vulnerabilidades con la severidad indicada. `format: table` produce una salida legible en los logs de GitHub Actions.

### Ejemplo de output de Trivy

```text
2024-01-15T10:30:00Z INFO Vulnerability scanning is enabled
2024-01-15T10:30:05Z INFO Detected OS: debian 12.4

ghcr.io/rush-vzla/rush-api:sha-abc1234 (debian 12.4)
=======================================================
Total: 0 (CRITICAL: 0)

Node.js (node-pkg)
==================
Total: 2 (CRITICAL: 0, HIGH: 1, MEDIUM: 1)

┌────────────────┬────────────┬──────────┬──────────────────────────┐
│    Library     │    CVE     │ Severity │         Title            │
├────────────────┼────────────┼──────────┼──────────────────────────┤
│ some-lib@1.2.3 │ CVE-2024-X │ HIGH     │ Prototype pollution in...│
└────────────────┴────────────┴──────────┴──────────────────────────┘
```

CRITICAL: 0 significa que el pipeline pasa. El HIGH queda en el reporte para revisión manual.

### ¿Qué hacer con una vulnerabilidad HIGH?

El proceso en Rush cuando aparece una HIGH:

1. Verificar si la versión afectada tiene una actualización disponible. Si la hay, actualizar.
2. Si no hay actualización disponible, evaluar si Rush usa la funcionalidad afectada.
3. Si no aplica: agregar al `.trivyignore` con comentario explicativo.
4. Si sí aplica y no hay fix: escalar al founder y documentar el riesgo aceptado.

### Archivo .trivyignore con documentación

```text
# CVE-2024-12345
# Librería: some-old-dep@1.0.0
# Motivo: vulnerability affects XML parsing, Rush does not use XML anywhere.
# Revisión pendiente: cuando haya nueva versión disponible (issue #234).
# Aprobado por: founder@somosrush.io — 2026-05-18
CVE-2024-12345
```

El comentario con fecha y aprobación convierte la excepción en algo auditable.

### Escaneo local antes de pushear

Para correr Trivy localmente antes de abrir un PR:

```bash
trivy image --severity CRITICAL,HIGH ghcr.io/rush-vzla/rush-api:local
```

Instalar Trivy en macOS:

```bash
brew install trivy
```
